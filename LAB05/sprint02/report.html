LAB05 — GraphQL vs REST: Relatório Final

Disciplina: Laboratório de Experimentação de Software Curso: Engenharia de Software — PUC Minas Gerado em: 24/06/2026 20:10

1. Introdução

A linguagem de consulta GraphQL, proposta pelo Facebook como alternativa às APIs REST, permite que clientes solicitem exatamente os campos que precisam — eliminando o over-fetching característico do modelo REST, onde endpoints retornam objetos completos independentemente da necessidade do consumidor. Vários sistemas passaram a oferecer ambas as abordagens simultaneamente, mas os benefícios mensuráveis da adoção do GraphQL ainda carecem de evidência empírica sólida e sistematizada.

Este experimento controlado busca responder objetivamente:

  • RQ1: Respostas a consultas GraphQL são mais rápidas que respostas a consultas REST?
  • RQ2: Respostas a consultas GraphQL têm tamanho menor que respostas a consultas REST?

Hipóteses

RQ1 — Tempo de Resposta (teste bicaudal)
H₀¹: mediana(TGraphQL) = mediana(TREST)
H₁¹: mediana(TGraphQL) ≠ mediana(TREST)
RQ2 — Tamanho da Resposta (teste unicaudal)
H₀²: mediana(SGraphQL) ≥ mediana(SREST)
H₁²: mediana(SGraphQL) < mediana(SREST)

2. Metodologia

Plataforma e Objetos Experimentais

A API pública do GitHub foi escolhida por disponibilizar simultaneamente REST (v3) e GraphQL (v4) sobre os mesmos dados, garantindo comparabilidade direta. Foram selecionados 20 repositórios populares, diversificados por linguagem e domínio: facebook/react, vuejs/vue, angular/angular, tensorflow/tensorflow, microsoft/vscode, flutter/flutter, kubernetes/kubernetes, golang/go, django/django, rails/rails, nodejs/node, rust-lang/rust, expressjs/express, pallets/flask, tiangolo/fastapi, spring-projects/spring-boot, laravel/laravel, pytorch/pytorch, scikit-learn/scikit-learn, hashicorp/terraform.

Tratamentos e Equivalência de Consultas

Desenho Experimental

Tipo: Experimento controlado within-subjects com medidas repetidas. Os mesmos 20 repositórios foram submetidos a ambos os tratamentos (REST e GraphQL), eliminando a variância inter-objeto como fator de confusão e aumentando o poder estatístico.
Randomização: A ordem REST→GraphQL foi sorteada aleatoriamente (50/50) em cada repetição para controlar efeitos de cache e sequenciamento.
Warm-up: 2 repetições iniciais por par foram descartadas para eliminar efeitos de cold-start (estabelecimento de conexão TCP, resolução DNS, etc.).
Volume: 20 repositórios × 4 tipos × 30 repetições × 2 APIs = 4.800 medições válidas esperadas.

Medição das Variáveis Dependentes

  • Tempo de resposta (s): time.perf_counter() antes e após o recebimento completo da resposta HTTP.
  • Tamanho da resposta (bytes): len(response.content) — comprimento do corpo bruto antes de qualquer descompressão adicional no cliente.

Testes Estatísticos

Utilizou-se o Teste de Wilcoxon Pareado (signed-rank), não-paramétrico e adequado para dados de tempo de resposta com distribuição assimétrica e cauda longa. Para controle do erro tipo I em múltiplos testes (4 tipos × 2 métricas = 8 comparações), aplicou-se a correção de Bonferroni: α_adj = 0,05 / 8 = 0.00625.

O tamanho de efeito reportado é a correlação rank-biserial r = (T⁺ − T⁻) / [n(n+1)/2], interpretada como: |r| < 0,1 = negligível; 0,1–0,3 = pequeno; 0,3–0,5 = médio; >0,5 = grande.

Ambiente de Execução

As medições foram realizadas a partir de uma máquina cliente com conexão doméstica à internet via protocolo HTTPS. Biblioteca cliente: Python requests. O cabeçalho Cache-Control: no-cache, no-store foi enviado em todas as requisições. O script aguardou automaticamente em caso de rate limiting da API do GitHub (headers X-RateLimit-Reset).

5. Discussão

RQ1 — Tempo de Resposta

A comparação dos tempos de resposta reflete o efeito combinado de dois fatores opostos: o GraphQL retorna menos dados (menor volume de transferência pela rede), mas o servidor precisa parsear a query e resolver o grafo de objetos — um overhead de processamento inexistente no modelo REST. O resultado líquido depende de qual fator domina em cada cenário de uso.

Para consultas simples de metadados (repo_info), onde a query GraphQL é pequena e o ganho de transferência é substancial (13 vs ~60 campos), a redução de payload pode se traduzir em menor tempo de transferência. Para listas com muitos itens (issues, pull_requests), o overhead de resolução do grafo no servidor pode compensar ou até superar o ganho de rede.

Diferenças pequenas ou não significativas no tempo são esperadas e alinhadas com a literatura: a vantagem do GraphQL em tempo tende a ser mais pronunciada em redes com largura de banda limitada, não em conexões de alta velocidade com servidores de infraestrutura como o GitHub.

RQ2 — Tamanho da Resposta

A redução de tamanho com GraphQL reflete diretamente a seleção de campos: enquanto o REST retorna objetos completos (incluindo URLs internas, permissões, metadados de paginação, etc.), o GraphQL retorna apenas os campos explicitamente solicitados na query. A diferença percentual esperada varia por tipo de consulta: repo_info apresenta a maior redução (13 vs ~60 campos), enquanto commits apresenta a menor (6 vs ~20 campos por item).

A compressão gzip habitual na API do GitHub reduz o impacto no nível de transporte, mas o ganho semântico permanece: menos campos processados pelo cliente e menos memória consumida no parsing.

Ameaças à Validade e Limitações

  • Especificidade de plataforma: Os resultados são válidos para a API do GitHub; outras implementações de GraphQL/REST podem ter perfis de desempenho diferentes (implementações customizadas tendem a ser mais leves que a infraestrutura global do GitHub).
  • Variação de rede: A latência cliente-servidor varia ao longo do tempo. A randomização da ordem e as medidas repetidas minimizam o viés, mas não eliminam completamente o ruído de rede.
  • Seleção de campos GraphQL: A escolha de quais campos solicitar afeta diretamente o tamanho e, indiretamente, o tempo. As queries foram calibradas para representar um uso típico — campos semanticamente equivalentes ao que um cliente extrairia da resposta REST.
  • Tipo de operação: Apenas operações de leitura simples foram testadas. Mutations e queries com múltiplos níveis de relacionamento podem apresentar diferenças mais pronunciadas.

Implicações Práticas

Para sistemas com restrições de largura de banda ou alto volume de requisições de leitura, a redução de tamanho do GraphQL tem valor direto e mensurável. Quanto ao tempo de resposta, a decisão entre GraphQL e REST deve considerar também a complexidade da implementação do servidor, os custos de operação e a curva de aprendizado das queries.
