Hipóteses Nula e Alternativa
Um par de hipóteses para cada pergunta de pesquisa
RQ1 · Tempo
Hipótese Nula H₀¹
mediana(TGraphQL) = mediana(TREST)

Não há diferença no tempo de resposta mediano entre GraphQL e REST. A adoção de um ou outro paradigma não impacta a velocidade das consultas.

Hipótese Alternativa H₁¹
mediana(TGraphQL) ≠ mediana(TREST)

Há diferença no tempo de resposta. O efeito não é trivialmente previsível: GraphQL retorna menos dados (↓ transferência) mas exige parsing de query no servidor (↑ overhead) — justificando hipótese bicaudal.

Bicaudal · α = 0,05
RQ2 · Tamanho
Hipótese Nula H₀²
mediana(SGraphQL) = mediana(SREST)

Não há diferença no tamanho mediano das respostas. Ambas as APIs retornam volumes equivalentes de dados para consultas semanticamente iguais.

Hipótese Alternativa H₁²
mediana(SGraphQL) < mediana(SREST)

GraphQL retorna respostas menores porque permite selecionar exatamente os campos desejados, eliminando o over-fetching estrutural do REST — que retorna objetos completos (40–60 campos) independente do que o cliente precisa. Hipótese unicaudal.

Unicaudal (menor) · α = 0,05

✓
Within-subjects (medidas repetidas)
Os mesmos 20 repositórios são submetidos a ambos os tratamentos (REST e GraphQL). Isso elimina a variância entre repositórios da comparação — cada repo é seu próprio controle, aumentando o poder estatístico sem aumentar o número de objetos.
✓
Comparação pareada
Para cada repetição, REST e GraphQL são executados na mesma janela temporal, gerando pares que o Wilcoxon analisa como diferenças intra-par — o ruído de latência de rede afeta os dois igualmente na mesma janela.
✓
Randomização da ordem
Em cada uma das 30 repetições, a ordem REST→GraphQL vs GraphQL→REST é sorteada com probabilidade 50/50. Isso neutraliza efeitos de aprendizagem, cache do servidor e quaisquer tendências temporais dentro de cada bloco de medição.
✓
Dois fatores de bloco
Bloco 1: tipo de consulta (repo_info, issues, PRs, commits) — análise segmentada detecta se REST/GQL se comportam diferente por complexidade da query.
Bloco 2: repositório — 20 repos com perfis distintos garantem generalização dos resultados.

H
Ameaças à Validade
Identificadas em quatro dimensões; controles aplicados onde possível
Variação de latência de rede
A latência entre cliente e servidor GitHub pode variar entre requisições independentes, tornando comparações não-pareadas injustas.
Comparação pareada e interleaved: REST e GraphQL medidos na mesma janela temporal, minimizando viés de tendências temporais
Cache do servidor
GitHub pode retornar respostas cacheadas para consultas repetidas, subestimando o tempo real de processamento e favorecendo artificialmente respostas mais rápidas.
Header Cache-Control: no-cache, no-store + Pragma: no-cache em todas as requisições
Rate limiting da API
GitHub limita a 5.000 req/h (REST) e 5.000 pontos/h (GraphQL). Atingir o limite interrompe a coleta e pode introduzir gaps temporais não controlados.
Detector automático de status 403/429; aguarda o reset via header X-RateLimit-Reset antes de continuar
Efeito de ordenação
Executar sempre REST antes de GraphQL (ou vice-versa) introduziria viés sistemático em favor do segundo tratamento (servidor em estado "aquecido").
Ordem (REST→GQL ou GQL→REST) sorteada aleatoriamente com p=0,5 em cada uma das 30 repetições
Cold-start de conexão TCP
As primeiras medições de cada par podem ser mais lentas por ausência de conexão TCP estabelecida e por efeitos de JIT no servidor GitHub.
Warm-up de 2 repetições descartadas por par antes de iniciar as 30 medições válidas
Especificidade de plataforma
Os resultados são válidos para a infraestrutura do GitHub. Implementações próprias de GraphQL/REST em servidores dedicados (ex.: BFF em Node.js) podem apresentar overhead relativo muito diferente. O GraphQL poderia ser proporcionalmente mais rápido em implementações leves ou mais lento em implementações mal otimizadas.
Seleção de repositórios populares
Os 20 repos são muito populares e ativos, com muitas issues e PRs abertas. Repositórios menores podem gerar respostas mais compactas em ambas as APIs, reduzindo as diferenças observadas. Os resultados podem não generalizar para o caso médio de repositórios do GitHub.
Tipo e complexidade das consultas
Foram selecionadas 4 consultas de leitura simples (single-request). Consultas de escrita (mutations), paginação profunda, ou queries com múltiplos níveis de relacionamento — onde o GraphQL é especialmente mais eficiente (elimina N+1 queries) — podem apresentar diferenças muito mais pronunciadas.
Compressão HTTP (gzip)
Se o servidor retornar Content-Encoding: gzip, o tamanho medido pelo cliente reflete o payload comprimido. Como gzip é mais eficiente em textos maiores e repetitivos, o REST (com mais campos de URL repetidos) pode ser relativamente favorecido, reduzindo a diferença de tamanho observada em relação ao dado semântico bruto.
Latência vs. throughput real
O tempo de resposta mede a latência de requisições sequenciais. Em sistemas de alta concorrência (múltiplas requisições paralelas), o impacto pode ser diferente — o GraphQL pode apresentar vantagem maior por reduzir o número total de bytes na rede, ou desvantagem por ter maior custo de parsing no servidor sob carga.
Equivalência semântica das queries
A seleção de quais campos solicitar no GraphQL afeta diretamente o tamanho da resposta. Escolher campos em excesso ou insuficientes distorceria a comparação. Os campos foram selecionados para representar o que um cliente típico extrairia da resposta REST correspondente — mas essa escolha é subjetiva e influencia o resultado de RQ2.
Não normalidade dos dados
Tempos de resposta de API costumam ter distribuição assimétrica com cauda longa — violando o pressuposto de normalidade dos testes paramétricos (t-test).
Uso de mediana (robusta a outliers) e teste de Wilcoxon pareado (não paramétrico, não assume normalidade)
Múltiplos testes simultâneos
São realizadas 4 tipos de consulta × 2 métricas = 8 comparações independentes, inflando a taxa de erro tipo I (falso positivo) se o nível α não for ajustado.
Reportar p-valores individuais + aplicar correção de Bonferroni (α_ajustado = 0,05/8 = 0,00625) para controlar FWER
